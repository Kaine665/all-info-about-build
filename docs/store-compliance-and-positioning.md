# 应用商店合规心智与上架叙事

可复用的原则层知识：平台规则、司法辖区法律与产品对外呈现之间的关系。适用于任何可能被归类为「敏感」或强监管类目的应用，不绑定某一具体产品名。

---

## 1. 应用商店不是「法外之地」

- 上架需遵守：**应用商店的开发者政策**（各商店不同），以及**应用分发地区的适用法律**。
- 二者可能同时适用：被拒或下架有时来自**平台规则**，有时来自**合规/权利主张**等链条。
- 含义：做产品规划时，应预留**政策与法律双重视角**，而不是只研究商店 UI 流程。

## 2. 「信息/工具」与「受监管的专业服务」

- 许多地区对**法律、医疗、金融**等专业服务有**准入或牌照**要求，与是否上架商店无直接关系。
- 若应用的名称、截图、描述或功能，使用户**合理认为**你们在提供**个案层面的专业意见或代理**，则合规风险通常高于**普适信息、计算器、模板、个人记录类工具**。
- **免责声明**有助于明确预期，但**一般不能替代**当地法律对「从事某类业务」的资质要求；具体界限需按业务与法域判断。

## 3. 审核偏严类目时的产品叙事策略（模板）

以下为主流「降低整包被拒概率」的通用思路，可按项目裁剪：

- **首版对外主叙事**：突出可验证、中性的工具价值（记录、统计、导出、流程说明等）。
- **强敏感或强依赖资质的能力**：可放在**后续版本**、**应用内较深入口**，或**独立子产品**，避免首屏与商店图触发过度联想。
- **UGC、地图、曝光类功能**：常单独触发内容与资质审核，宜与核心工具包**拆分评估**。

## 4. 知识分层（维护本仓库时）

- **项目专属事实**（某仓库技术栈、某功能清单）与**可复用原则**（分发、合规、叙事）宜分开存放，避免通用知识库被单一产品细节占满。

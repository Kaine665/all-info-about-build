# inbox（未整理信息信箱）

本目录用于**临时存放**与构建、分发、测试、环境相关的原始笔记、会议结论草稿、待决问题清单等。内容在整理并写入 `SPEC.md` / `README.md` 等正式文档前，先落在此处，避免散落在 issue 或个人笔记里。

## 契约（必须遵守）

1. **不放密钥与真实凭证**  
   禁止提交 Access Key、Secret Key、会话令牌、私有 URL 带签名参数、客户真实数据。可写「占位符名」或「从何处领取（内部链接用文字描述，不放可点击含秘文）」。

2. **每条记录自洽、可检索**  
   单文件尽量只谈一个主题；文件名见下文规范。正文写清**背景、结论或待决、下一步**之一即可，避免只有碎片词。

3. **标注状态与日期**  
   文件顶部使用 YAML frontmatter（见模板）。`status` 用于区分草稿与已吸收；已迁移到正式文档的条目应把 `status` 改为 `absorbed` 并在正文末行给出**目标文档与章节**。

4. **与正式文档的关系**  
   inbox 不是规范来源；**以仓库根目录的正式文档为准**。inbox 里与正式文档冲突时，以正式文档为准，并应修正 inbox 或标记 `obsolete`。

5. **生命周期**  
   - `draft`：刚扔进信箱，尚未评审。  
   - `open`：已确认仍有效、待决或待实现跟踪。  
   - `absorbed`：要点已写入 SPEC/README 等。  
   - `obsolete`：被新结论替代或不再适用（正文简短说明原因）。

## 文件命名

推荐：

```text
YYYY-MM-DD-<主题短名>.md
```

示例：`2026-04-12-cos-test-bucket.md`。同一天多文件时在短名后加后缀：`-2`、`-adb-install`。

## 正文建议结构（非强制，但利于对齐）

- **上下文**：一句话说明为何现在记这条。  
- **选项 / 事实**：列表即可。  
- **推荐（若有）**：谁倾向哪一选项、依据。  
- **待决**：需谁拍板、截止或触发条件。  
- **链接**：对外 issue/文档用完整 URL；内部敏感链接只写路径或代号。

## 模板

复制 `_TEMPLATE.md` 新建文件，填好 frontmatter 后再写正文。

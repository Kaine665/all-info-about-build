## 上下文

在已有 SPEC 类文档之外，从「能装、能测、能交付」倒推，有一批主题适合提前拍板，避免实现阶段各自假设不一致。下列从对话整理为收件箱备忘，供后续吸收进正式文档（根目录 `README` / `SPEC` 或外部知识库）。

## 正文

### 1. 安装与分发

- **给谁测**：仅本人 / 小范围 / 未来上架——路径不同。  
- **安装形态**：Android Studio 直连、`adb install`、Play 内测轨道、企业分发等。  
- **签名与渠道**：debug / release；是否多环境（测试桶 vs 正式）用不同包名或 build flavor。  
- **最低设备矩阵**：真机型号；是否必须覆盖某 OEM 录音机行为。

### 2. 测试：策略与可接受成本

- **分层**：单元（队列/DB）、仪器化（MediaStore/权限）、手工验收（如 SPEC 中若干条）各自占比或底线。  
- **COS 测试**：专用测试桶、子账号最小权限、CI 是否允许跑、密钥从哪来。  
- **网络与后台**：弱网、飞行模式、杀进程、省电策略对 WorkManager 的影响；谁测、用什么清单或脚本。  
- **日期/时区**：自动化里如何模拟「当日」；难则是否接受以手工为主。

### 3. 环境与数据

- **凭证**：开发机 Secret 管理；是否禁止真实密钥进仓库或 issue。  
- **测试数据**：合成录音 vs 真机录音；空列表、超大文件、写入中的文件如何构造。

### 4. 产品与边界（在 SPEC 上再钉一层）

- **通知**：失败/重试是否进通知栏、是否第一期。  
- **隐私与文案**：首次保存密钥的风险提示粒度；是否要有导出/清除配置。  
- **与 Non-goals 的灰区**：例如仅 MediaStore 遇厂商路径异常时，是提示放弃还是失败可重试。

### 5. 工程与维护

- **技术栈**：Kotlin、最低/目标 SDK、Compose 等（SPEC 已定部分可一次定稿）。  
- **可观测性**：崩溃收集、日志级别（尤其不得打印 SecretKey）。  
- **版本与更新**：第一期是否仅侧载、版本号规则。

### 6. 协作方式

- 待决问题短表：选项 A/B、推荐、负责人、写入 README 还是 SPEC。  
- 安装与测试可单独成 README 一节：如何构建 APK、连测试桶、发布前检查。

### 7. 收窄约束（可选起点）

- 例如：**第一期仅自用 + 一个测试桶**，可自然收窄安装、COS 与分发方案。

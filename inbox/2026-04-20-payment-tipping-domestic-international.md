## 上下文

本文整理公开站点场景下的收款、打赏、会员与一次性付款方案对比（国内 / 国外），供个人创作者、独立开发者及小型项目团队选型参考。正文含免责声明与官方来源链接；上线与报税等事项以当地法规及平台现行条款为准。

## 正文

更新日期：2026-04-20

适用对象：个人创作者、独立开发者、小型项目团队、作品站 / 项目站运营者

说明：
- 本文整理的是此前讨论过的收款与打赏方案，重点关注“公开放在网站上收支持 / 打赏 / 会员费 / 一次性付款”场景。
- 本文不是法律、税务或会计意见。正式上线前，仍应以平台最新条款、支付机构要求和所在地区税务规则为准。
- 费用、提现周期、支持国家和风控规则可能会变，请在实际使用前再次核对官方页面。

## 一页结论

如果只看当前最实用的选择：

- 国内中文用户为主：首选 `爱发电`
- 海外用户为主：首选 `Ko-fi`
- 主要面对开发者 / 开源用户：补充 `GitHub Sponsors`
- 想自己掌控品牌和支付页：考虑 `Stripe Payment Links`
- 只想快速做一个海外轻量打赏页：`Buy Me a Coffee`
- 想长期经营订阅制社群：`Patreon`
- 不建议把 `个人静态收款码` 直接长期公开挂在网站首页当主收款方式

### 这次决策里为什么没有把 Buy Me a Coffee 和 Stripe 作为主推

这是本次决策的明确前提之一：

- 当前主体位于 `中国大陆` 时，`Stripe` 官方支持地区列表不包含中国大陆
- `Buy Me a Coffee` 的创作者打款依赖 `Stripe`
- `Buy Me a Coffee` 官方的创作者打款支持国家列表中也没有中国大陆

因此，对“主体在中国大陆、希望正规长期收款”的场景：

- `Stripe Payment Links` 暂时不适合作为主收款方案
- `Buy Me a Coffee` 也不适合作为主收款方案
- 它们不是完全不能访问，而是“创作者开户 / 打款落地”这一层存在现实障碍

所以这份文档里仍保留它们作为国际方案备选，但没有把它们列为当前场景下的主推荐。

## 怎么选

### 如果你是个人创作者

- 想快速开始国内打赏：`爱发电`
- 想快速开始海外打赏：`Ko-fi`
- 你的受众主要是程序员 / GitHub 用户：`GitHub Sponsors + Ko-fi`
- 你只想放一个非常简单的海外支持页：`Buy Me a Coffee`

### 如果你是项目站 / 团队

- 中文项目：`爱发电` 或 `支付宝 / 微信商家收款`
- 国际项目：`Stripe Payment Links`
- 开源团队：`GitHub Sponsors`
- 长期做会员社群：`Patreon`

## 总览表

| 方案 | 适合谁 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| 直接个人收款码 | 熟人、小范围、临时收款 | 快、门槛最低 | 合规边界最敏感，公开网页长期挂码不稳，难做对账和说明 | 不建议做公开站主方案 |
| 支付宝 / 微信商家收款 | 国内个体、项目、小团队 | 用户熟悉，国内支付转化高 | 更偏经营收款，不是“创作者平台”，内容和会员能力弱 | 适合作为国内补充方案 |
| 爱发电 | 中文创作者、独立项目 | 场景契合，中文用户理解成本低，会员和更新机制成熟 | 平台抽成固定，提现节奏不是实时 | 国内主推 |
| Ko-fi | 海外创作者、独立开发者 | 轻量、可嵌站、功能全、对一次性支持友好 | 需要 PayPal 或 Stripe，支付与争议处理更分散 | 海外主推 |
| Buy Me a Coffee | 想极简上线海外打赏页的人 | 上手快、页面简单、无月费 | 功能深度和自主控制感略弱于 Ko-fi | 海外轻量备选 |
| Patreon | 长期做订阅制、会员社群的人 | 会员体系成熟、社区工具强 | 费率高于 Ko-fi / BMC，运营更重 | 订阅制强选项 |
| GitHub Sponsors | 开源项目、开发者 | GitHub 场景天然契合，个人赞助零平台费 | 用户群更窄，不适合泛内容创作者 | 开源强补充 |
| Stripe Payment Links | 想自己掌控品牌和支付页的人 | 无代码、适合一次性 / 订阅 / donation，和自站整合自然 | 你自己承担更多税务、风控、退款与配置工作 | 品牌化强，但更“自己做系统” |

## 国内方案

### 1. 直接个人收款码

适合：
- 熟人、小范围、临时支持
- 线下当面收款

优点：
- 最快上线
- 不需要额外平台
- 对已经认识你的人来说操作成本最低

缺点：
- 对公开网站场景不稳，尤其是长期、远程、非面对面收款
- 账目说明、导出、对账、会员权益、退款机制都弱
- 用户难以判断这是“赞赏”还是“经营收款”
- 一旦金额上来，更容易触发风控、合规和税务问题

合规提醒：
- 中国人民银行通知明确要求：对具有明显经营活动特征的个人，不得继续通过个人收款条码提供经营活动相关收款服务。
- 同时，个人静态收款条码原则上禁止用于远程非面对面收款。

适合结论：
- 适合临时或熟人收款
- 不适合当作公开网站的长期主收款方式

### 2. 支付宝 / 微信商家收款

适合：
- 国内个人经营者、小团队、项目方
- 需要国内用户低门槛付款

优点：
- 国内用户最熟悉，支付转化高
- 比个人码更适合公开经营或长期收款
- 如果你本来就有经营收款需求，这条路更顺

缺点：
- 这本质上更像“经营收款工具”，不是“创作者支持平台”
- 内容更新、会员分层、支持者关系管理能力较弱
- 公开站上如果你需要“为什么支持我”“支持后能看到什么”，表达链路不如爱发电清晰
- 具体费率和开通要求依赖平台、行业和账户情况，变动空间较大

适合结论：
- 适合作为国内支付补充
- 如果你的场景不是纯打赏，而是“项目支持 + 偶尔卖内容 / 服务”，它会比个人码稳

### 3. 爱发电

适合：
- 中文创作者
- 独立开发者、写作者、设计师、内容创作者
- 想做赞赏、会员、更新页、档位支持

优点：
- 中文场景适配最好，用户一看就知道是“支持创作者”
- 支持赞助、会员档位、更新动态、开发者接入
- 官方 FAQ 明确费用结构，创作者拿到比例清楚
- 对中文用户来说认知成本比 Ko-fi、Patreon 更低

缺点：
- 平台抽成固定
- 钱不是实时直接进你自己的支付账户，而是按平台节奏入账、提现
- 面向国际用户时认知和支付体验不如 Ko-fi / Stripe

费用与节奏：
- 爱发电官方 FAQ：平台收取赞助金额的 `6%`，创作者获得 `94%`
- 其中 `1%` 是支付宝、微信等支付通道手续费，`5%` 是平台服务费
- 当月赞助会在次月 1 日白天进入余额，再提现

适合结论：
- 如果你的主要受众是中文用户，它通常是国内最平衡的主方案

## 国外方案

### 4. Ko-fi

适合：
- 海外支持者
- 独立开发者、创作者、个人项目站
- 想做“一次性打赏 + 月度支持 + 商店 + commissions”的人

优点：
- 很轻，适合先从打赏开始
- 页面、按钮、嵌入网站都很方便
- 支持一次性支持、月度支持、会员、商店、委托等
- 钱直接进入你自己的 `PayPal` 或 `Stripe`，平台不代持
- 对个人站、作品站、开发者页面都很友好

缺点：
- 你需要自己处理更多支付账户、退款、争议相关流程
- 如果你使用 PayPal / Stripe，有些国家或账户类型会带来额外摩擦
- 对中国大陆用户并不天然友好，更多适合海外支付场景

费用：
- Ko-fi Help 当前说明：
  - 免费模式：一次性 Tips `0%`，其他如 Memberships、Monthly payments、Commissions、Shop 为 `5%`
  - Contributor：所有支付类型 `5%`
  - Ko-fi Gold：`$12 / 月`，可降为 `0%` 服务费
- 支付处理费仍由 `PayPal` / `Stripe` 收取，Ko-fi 文档示例写的是通常约 `3% + $0.30`

适合结论：
- 如果你想做一个轻量但不“简陋”的海外支持页，`Ko-fi` 是最均衡的选择

### 5. Buy Me a Coffee

适合：
- 想要最快速上线海外打赏页的人
- 不想一开始配置太多东西的人

优点：
- 设置快，官方写明大多数创作者可在 `5 分钟内` 发布页面
- 支持一次性支持、会员、Shop、按钮嵌入
- 对外表达简单直接，支持者无需注册也能一次性付款
- 页面干净，适合轻运营

缺点：
- 长期来看，功能深度和“工具箱感”通常不如 Ko-fi
- 主要依赖 Stripe 打款
- 对“把它当成自己的品牌化支付层”这件事，控制感弱于 Stripe Payment Links
- 对中国大陆主体不友好。创作者打款依赖 Stripe，而官方打款支持国家列表中没有中国大陆

费用与打款：
- 平台收 `5%`
- 信用卡处理费通常约 `2.9% + $0.30`
- 账户达到 `$10` 后会审核，通过后每周三打到你的 Stripe，之后再由 Stripe 打到银行账户

适合结论：
- 如果你的重点是“现在就上线，不想折腾”，它比 Patreon 更轻，比 Stripe 更省事
- 如果你的主体在中国大陆，它不应被视为当前阶段的主方案

### 6. Patreon

适合：
- 长期做订阅制、会员社群
- 想持续输出会员内容、社区互动、分层权益的人

优点：
- 会员体系成熟
- 社区工具和付费内容能力强
- 如果你真的在经营持续订阅，而不是单次打赏，它比大多数平台更系统

缺点：
- 新创作者当前标准平台费较高
- 支付处理费、货币转换费、提现费叠加后，综合成本明显高于 Ko-fi / GitHub Sponsors
- 平台气质更偏“会员运营”，如果你只想收一点支持，会显得偏重

费用：
- Patreon Help 当前说明：2025-08-04 之后发布页面的新创作者，标准平台费为 `10%`
- 处理费依支付方式和币种而变
  - 例如 USD payout 标准费率中，信用卡 / Apple Pay 为 `2.9% + $0.30`
  - 非同币种支付还可能有 `2.5%` 货币转换费
- 提现时还有 payout transaction fee

适合结论：
- 只有在你明确要做“长期会员制经营”时，它才值得优先考虑

### 7. GitHub Sponsors

适合：
- 开源维护者
- 个人开发者
- 受众主要是 GitHub 用户、技术用户的人

优点：
- 场景天然契合开发者和开源项目
- 可以直接挂在 GitHub 个人主页和仓库 Sponsor 按钮
- 官方当前说明：来自个人账户的赞助 `0 平台费`，`100%` 到开发者或组织
- 支持一次性和月度 tiers

缺点：
- 用户群更窄，不适合泛内容、非技术受众
- 有地区支持限制
- 更适合开源与技术赞助，不如 Ko-fi 那么通用

费用：
- GitHub Docs 当前说明：
  - 个人账户发起赞助：`0` 费用
  - 组织账户发起赞助：最高 `6%`
  - 其中 `3%` 信用卡处理费 + `3%` GitHub 服务费

适合结论：
- 如果你做的是开源、工具、库、技术内容，它几乎应该作为必选补充方案

### 8. Stripe Payment Links

适合：
- 想在自己网站里维持品牌统一
- 不想依赖“创作者平台主页”作为主阵地
- 想收一次性付款、订阅费或 donation

优点：
- 无代码生成支付链接
- 可以卖产品、服务、订阅，也可以收 donation
- 非常适合和自有网站、邮件、社媒直接整合
- 品牌感强，看起来像你自己的支付页，而不是某个平台主页

缺点：
- 你自己承担更多运营职责：税务、对账、风控、退款策略、收款说明
- 对创作者社区、会员内容、支持者关系管理几乎没有平台加成
- 某些国家 / 地区开户和打款支持有门槛
- 对中国大陆主体不适配。当前官方支持地区列表中不包含中国大陆

费用：
- Stripe 官方目前将 Payment Links 归入 Payments 定价
- 页面当前展示的标准基线通常是 `2.9% + 30¢` 起
- 订阅则会叠加 Stripe Billing 定价
- 国际卡、货币转换等还可能增加额外成本

适合结论：
- 如果你更像在“运营一个产品或品牌网站”，而不是“运营一个创作者主页”，它会比 Ko-fi 更专业
- 但如果你的主体在中国大陆，这个方案当前更多是参考项，不是落地项

## 我会怎么推荐

### 场景 1：个人作品站 / 独立开发者

推荐组合：
- 中文受众：`爱发电`
- 海外受众：`Ko-fi`
- 开源用户：`GitHub Sponsors`

原因：
- 这组组合覆盖面够大
- 平台都比较容易理解
- 你不需要自己先搭一整套支付系统

### 场景 2：开源项目

推荐组合：
- 主：`GitHub Sponsors`
- 辅：`Ko-fi`

原因：
- GitHub Sponsors 对技术受众最自然
- Ko-fi 用来承接非 GitHub 场景和一次性支持

### 场景 3：纯海外支持页

推荐组合：
- 轻量：`Ko-fi`
- 更极简：`Buy Me a Coffee`
- 更品牌化：`Stripe Payment Links`

### 场景 4：长期会员制社群

推荐组合：
- 主：`Patreon`
- 辅：`Ko-fi`

### 场景 5：国内公开网站想放收款入口

推荐组合：
- 主：`爱发电`
- 辅：`支付宝 / 微信商家收款`

不建议：
- 直接长期公开挂 `个人静态收款码`

## 个人建议

如果你现在要给“职业墓场”这类项目站配置收款入口，我会优先这样做：

1. 国内页先放 `爱发电`
2. 海外页放 `Ko-fi`
3. 如果项目或作者本身有开源仓库，再加 `GitHub Sponsors`
4. 等项目确实需要更强品牌控制时，再考虑 `Stripe Payment Links`

这样做的好处是：
- 先把可用性做出来
- 先降低合规和支付摩擦
- 不会一开始就把自己拖进税务、退款、支付风控和系统集成细节里
